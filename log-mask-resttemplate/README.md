# Log Mask RestTemplate Common Guide

**English** | [简体中文](README.zh-CN.md)

This guide covers the log structure, content switches, and masking configuration shared by the Spring Boot 2 and Spring Boot 3 RestTemplate Starters. See the corresponding Starter guide for dependency coordinates, Java versions, package names, and RestTemplate selection:

- [Spring Boot 2 Starter guide](../log-mask-resttemplate-spring-boot2-starter/README.md)
- [Spring Boot 3 Starter guide](../log-mask-resttemplate-spring-boot3-starter/README.md)

## Log output

Call logs use the `log.mask.http` logger. When `INFO` is enabled, each HTTP call produces at most one single-line JSON entry containing both the request and response:

```yaml
logging:
  level:
    log.mask.http: INFO
```

When this logger does not accept `INFO`, Log Mask neither collects nor writes call content.

### Global switches

| Property | Default | Effect |
| --- | --- | --- |
| `log-mask.logging.rest-template.enabled` | `true` | Master switch for RestTemplate call logs; when disabled, logging capability is not installed and call content is neither collected nor written |
| `logging.level.log.mask.http` | Determined by the application's logging configuration | Set to `INFO` to enable call logs, or `OFF` to stop collection and output |
| `log-mask.governance.enabled` | `true` | Whether to apply masking and exclusion rules to Path, Query, Header, and Body content |

Setting `log-mask.governance.enabled=false` does not disable call logs. It skips every masking and exclusion rule, so collected values enter the log unchanged.

Path, Query, and Header values without a matching rule are also logged unchanged, as are Body fields without `@Mask` or `@LogExclude`.

### Appenders

With Spring Boot's default logging configuration, call logs are written to the console. To write them to a separate JSON Lines file, configure a dedicated appender for `log.mask.http`.

Logback example:

```xml
<appender name="LOG_MASK_HTTP" class="ch.qos.logback.core.FileAppender">
    <file>logs/log-mask-http.jsonl</file>
    <append>true</append>
    <encoder>
        <pattern>%msg%n</pattern>
        <charset>UTF-8</charset>
    </encoder>
</appender>

<logger name="log.mask.http" level="INFO" additivity="false">
    <appender-ref ref="LOG_MASK_HTTP" />
</logger>
```

Log4j2 example:

```xml
<File name="LogMaskHttp" fileName="logs/log-mask-http.jsonl">
    <PatternLayout pattern="%m%n" charset="UTF-8" />
</File>

<Logger name="log.mask.http" level="info" additivity="false">
    <AppenderRef ref="LogMaskHttp" />
</Logger>
```

Log Mask does not create appenders or modify the root logger.

## Log structure

With the default `STANDARD` structure, each key has the following meaning:

```text
{}                                  # Root object of an HTTP call log
├── event                           # Always http_exchange
├── schemaVersion                   # Log schema version; currently 1
├── timestamp                       # UTC time when the call ended and the log was prepared
├── exchangeId                      # Unique identifier generated by Log Mask for this HTTP call
├── traceId                         # Trace identifier read from MDC, or null when unavailable
├── durationMs                      # Time spent in the downstream interceptor chain, in milliseconds
├── governanceEnabled               # Whether masking and exclusion rules were enabled for this call
├── request                         # Request log
│   ├── method                      # HTTP request method
│   ├── uriState                    # Processing result for URI log content
│   ├── uri                         # Processed request URI
│   │   ├── full                    # Processed full URL; always present
│   │   ├── scheme                  # Lowercase URI scheme
│   │   ├── host                    # Lowercase target host name
│   │   ├── port                    # Effective target port, or null when it cannot be determined
│   │   ├── path                    # Processed path with percent encoding preserved
│   │   └── query                   # Processed query parameter entries
│   │       └── entry               # One query parameter
│   │           ├── name            # Query parameter name
│   │           └── values          # All values for that parameter
│   ├── headersState                # Processing result for request header log content
│   ├── headers                     # Processed request header entries
│   │   └── entry                   # One request header
│   │       ├── name                # Lowercase header name
│   │       └── values              # All values for that header
│   ├── bodyState                   # Processing result for request body log content
│   └── body                        # Processed request body
└── response                        # Response log, or null when no response was obtained
    ├── status                      # HTTP response status code
    ├── headersState                # Processing result for response header log content
    ├── headers                     # Processed response header entries
    │   └── entry                   # One response header
    │       ├── name                # Lowercase header name
    │       └── values              # All values for that header
    ├── bodyState                   # Processing result for response body log content
    └── body                        # Processed response body
```

If the request was sent but no response was obtained, `response` is `null` and the original business exception continues to the caller.

`uriState`, `headersState`, and `bodyState` can have these values:

| State | Applies to | Meaning |
| --- | --- | --- |
| `SUCCESS` | URI, Header, Body | Log content was generated successfully |
| `FALLBACK_APPLIED` | URI, Header | A rule failed and the affected value was replaced with fixed fallback content |
| `LIMIT_EXCEEDED` | Body | The Body exceeded the maximum log size |
| `PROCESSING_FAILED` | URI | The URI could not be processed completely, so fallback content was written |
| `PROCESSING_FAILED` | Body | Body log content could not be generated, so Body is an empty string |
| `DISABLED` | Header, Body | The corresponding content was disabled by configuration |

`durationMs` starts when the Log Mask interceptor enters the downstream interceptor chain and ends when a response is obtained or an exception occurs. It includes authentication, retries, backoff, and network calls, but excludes request and response converter processing.

## Path

The HTTP method and full URL are always logged and cannot be disabled separately. `uri.details-enabled` only controls the separate `scheme`, `host`, `port`, `path`, and `query` fields. When it is disabled, `uri.full` remains and Path and Query rules still process the full URL.

### Switches and parameters

| Property | Default | Effect |
| --- | --- | --- |
| `log-mask.logging.rest-template.uri.details-enabled` | `true` | Whether to include `scheme`, `host`, `port`, `path`, and `query` in addition to the full URL |
| `log-mask.governance.http.path.rules` | `[]` | Path masking rules; an empty list leaves every Path segment unchanged |

Each Path rule contains:

| Parameter | Required | Description |
| --- | --- | --- |
| `pattern` | Yes | Absolute Path template, such as `/customers/{id}` |
| `host` | No | Apply only to this host; matching is case-insensitive |
| `method` | No | Apply only to this HTTP method; matching is case-insensitive |
| `variables[].name` | Yes | Name matching the corresponding `{name}` in the template |
| `variables[].type` | Either this or `type-code` | Built-in masking type |
| `variables[].type-code` | Either this or `type` | Custom masking strategy code |

Built-in types are `REDACT`, `PHONE`, `EMAIL`, `ID_CARD`, `BANK_CARD`, and `FULL`. Path parameters do not support `EXCLUDE`. Every `{name}` in a template must have a matching variable configuration. `*` matches one Path segment that remains unchanged; `**` is not supported.

### FULL example

```yaml
log-mask:
  governance:
    http:
      path:
        rules:
          - pattern: /customers/{id}
            variables:
              - name: id
                type: FULL
```

Actual request:

```text
https://api.example.com/customers/ABC
```

URI in the log:

```json
{
  "full": "https://api.example.com/customers/%2A%2A%2A",
  "scheme": "https",
  "host": "api.example.com",
  "port": 443,
  "path": "/customers/%2A%2A%2A",
  "query": []
}
```

`FULL` replaces `ABC` with the same-length value `***`; asterisks are encoded as `%2A%2A%2A` in the URL. The actual request URI is unchanged.

## Query

Query has no separate disable switch. The full URL always includes Query content. Setting `uri.details-enabled=false` removes only the separate `request.uri.query` field, while Query rules continue to process `request.uri.full`.

### Switches and parameters

| Property | Default | Effect |
| --- | --- | --- |
| `log-mask.logging.rest-template.uri.details-enabled` | `true` | Whether to include the structured `request.uri.query` field |
| `log-mask.logging.rest-template.name-value-shape` | `STANDARD` | JSON shape for Query and Header values; can be changed to `COMPACT` |
| `log-mask.governance.http.query.rules` | `[]` | Query masking and exclusion rules; an empty list records every parameter unchanged |

Each Query rule contains:

| Parameter | Required | Description |
| --- | --- | --- |
| `name` | Yes | Query parameter name, matched exactly and case-sensitively after UTF-8 decoding |
| `host` | No | Apply only to this host; matching is case-insensitive |
| `type` | Either this or `type-code` | Built-in masking or exclusion type |
| `type-code` | Either this or `type` | Custom masking strategy code |

Built-in types are `REDACT`, `EXCLUDE`, `PHONE`, `EMAIL`, `ID_CARD`, `BANK_CARD`, and `FULL`. Query rules cannot be limited by HTTP method or Path.

### FULL example

```yaml
log-mask:
  governance:
    http:
      query:
        rules:
          - name: token
            type: FULL
```

Actual request:

```text
https://api.example.com/search?token=ABC&visible=ok
```

URI in the log:

```json
{
  "full": "https://api.example.com/search?token=%2A%2A%2A&visible=ok",
  "scheme": "https",
  "host": "api.example.com",
  "port": 443,
  "path": "/search",
  "query": [
    {"name":"token","values":["***"]},
    {"name":"visible","values":["ok"]}
  ]
}
```

`visible` has no matching rule and therefore remains unchanged. The actual request parameters are not modified.

### Query shape

`STANDARD` writes an array:

```json
[
  {"name":"token","values":["***"]}
]
```

With `COMPACT`, it writes an object:

```yaml
log-mask:
  logging:
    rest-template:
      name-value-shape: COMPACT
```

```json
{
  "token": ["***"]
}
```

This setting also changes the shape of Request Header and Response Header values.

## Request Header

Request Header logging is enabled by default and can be disabled independently. When disabled, `request.headersState` is `DISABLED`, `request.headers` is `null`, and all other request and response content continues to be logged.

### Switches and parameters

| Property | Default | Effect |
| --- | --- | --- |
| `log-mask.logging.rest-template.request.headers-enabled` | `true` | Whether to log Request Headers |
| `log-mask.logging.rest-template.name-value-shape` | `STANDARD` | JSON shape for Header values; can be changed to `COMPACT` |
| `log-mask.governance.http.headers.request.rules` | `[]` | Request Header masking and exclusion rules; an empty list records every Header unchanged |

Each Request Header rule contains:

| Parameter | Required | Description |
| --- | --- | --- |
| `name` | Yes | Header name; matching is case-insensitive |
| `host` | No | Apply only to this host; matching is case-insensitive |
| `type` | Either this or `type-code` | Built-in masking or exclusion type |
| `type-code` | Either this or `type` | Custom masking strategy code |

Built-in types are `REDACT`, `EXCLUDE`, `PHONE`, `EMAIL`, `ID_CARD`, `BANK_CARD`, and `FULL`. Multi-value Headers are processed one value at a time, and Header names are lowercase in the log.

Request Headers are captured before Log Mask enters the downstream interceptor chain. Headers added or changed by later interceptors, such as an authentication interceptor, do not appear in this call log.

### FULL example

```yaml
log-mask:
  governance:
    http:
      headers:
        request:
          rules:
            - name: X-Api-Key
              type: FULL
```

Actual Request Header:

```text
X-Api-Key: ABC
```

`request.headers` in the log:

```json
[
  {"name":"x-api-key","values":["***"]}
]
```

The actual Request Header is unchanged.

## Request Body

Request Body logging is enabled by default and can be disabled independently. When disabled, `request.bodyState` is `DISABLED`, `request.body` is an empty string, and all other request and response content continues to be logged.

### Switches and parameters

| Property or rule | Default | Effect |
| --- | --- | --- |
| `log-mask.logging.rest-template.request.body-enabled` | `true` | Whether to log the Request Body |
| `log-mask.logging.rest-template.max-body-size` | `64KB` | Maximum log size for one Request Body, calculated separately from the Response Body |
| `@Mask` | None | Apply a built-in or custom masking strategy to a field of a specific Java type |
| `@LogExclude` | None | Remove a field from the logged Body while retaining it in the actual request |

`max-body-size` accepts values from 1 to 2,147,483,647 bytes. When the Body exceeds this limit, `request.bodyState` is `LIMIT_EXCEEDED`, `request.body` is an empty string, and no truncated content is logged.

### FULL example

```java
import io.github.summerwenlabs.log.mask.governance.LogExclude;
import io.github.summerwenlabs.log.mask.governance.Mask;
import io.github.summerwenlabs.log.mask.governance.MaskType;

public final class CredentialRequest {

    @Mask(type = MaskType.FULL)
    public String customerCode;

    @LogExclude
    public String accessToken;
}
```

Actual Request Body:

```json
{
  "customerCode": "ABC",
  "accessToken": "TOKEN-EXAMPLE-ABCD"
}
```

`request.body` in the log:

```json
{
  "customerCode": "***"
}
```

The actual request and original application object are unchanged.

### Body types

| Request Body type | Log behavior |
| --- | --- |
| Application object with a specific Java type | Generate log JSON using `@Mask`, `@LogExclude`, and custom strategies; field rules continue to apply within nested objects, collections, arrays, and application objects stored in Maps |
| `String` | Record as a plain JSON string without parsing its content as JSON again |
| `byte[]` | Encode as a JSON string using Jackson Base64 encoding |
| Content written by another converter | Decode using the declared charset, or strict UTF-8 when no charset is declared; use Base64 when decoding fails, without applying field-level rules |
| No Request Body | Record JSON `null` |

## Response Header

Response Header logging is enabled by default and can be disabled independently. When disabled, `response.headersState` is `DISABLED`, `response.headers` is `null`, and all other request and response content continues to be logged.

### Switches and parameters

| Property | Default | Effect |
| --- | --- | --- |
| `log-mask.logging.rest-template.response.headers-enabled` | `true` | Whether to log Response Headers |
| `log-mask.logging.rest-template.name-value-shape` | `STANDARD` | JSON shape for Header values; can be changed to `COMPACT` |
| `log-mask.governance.http.headers.response.rules` | `[]` | Response Header masking and exclusion rules; an empty list records every Header unchanged |

Each Response Header rule uses the same `name`, `host`, `type`, and `type-code` parameters as a Request Header rule, but requests and responses have independent rule sets.

### FULL example

```yaml
log-mask:
  governance:
    http:
      headers:
        response:
          rules:
            - name: X-Response-Key
              type: FULL
```

Actual Response Header:

```text
X-Response-Key: ABC
```

`response.headers` in the log:

```json
[
  {"name":"x-response-key","values":["***"]}
]
```

The actual Response Header is unchanged.

## Response Body

Response Body logging is enabled by default and can be disabled independently. When disabled, `response.bodyState` is `DISABLED`, `response.body` is an empty string, and all other request and response content continues to be logged.

### Switches and parameters

| Property or rule | Default | Effect |
| --- | --- | --- |
| `log-mask.logging.rest-template.response.body-enabled` | `true` | Whether to log the Response Body |
| `log-mask.logging.rest-template.max-body-size` | `64KB` | Maximum log size for one Response Body, calculated separately from the Request Body |
| `@Mask` | None | Apply a built-in or custom masking strategy to a field of a specific Java type |
| `@LogExclude` | None | Remove a field from the logged Body while retaining it in the response object returned to application code |

When the Body exceeds the limit, `response.bodyState` is `LIMIT_EXCEEDED`, `response.body` is an empty string, and no truncated content is logged.

### FULL example

```java
import io.github.summerwenlabs.log.mask.governance.LogExclude;
import io.github.summerwenlabs.log.mask.governance.Mask;
import io.github.summerwenlabs.log.mask.governance.MaskType;

public final class CredentialResponse {

    @Mask(type = MaskType.FULL)
    public String customerCode;

    @LogExclude
    public String accessToken;
}
```

Actual Response Body:

```json
{
  "customerCode": "ABC",
  "accessToken": "TOKEN-EXAMPLE-ABCD"
}
```

`response.body` in the log:

```json
{
  "customerCode": "***"
}
```

The response object returned to application code is unchanged.

### Body types

| Response Body type | Log behavior |
| --- | --- |
| Successful response with a specific Java type | Generate log JSON from the application object produced by normal deserialization and apply its field rules |
| `String` | Record as a plain JSON string without parsing its content as JSON again |
| `byte[]` | Encode as a JSON string using Jackson Base64 encoding |
| Error response or another untyped response | Record only bytes actually read by the business flow; write a JSON string when they can be decoded with the charset, or Base64 when they cannot |
| No Response Body | Record JSON `null` |

Log Mask does not pre-read or complete-read an error response for logging. Content the business flow does not read is not written to the log. If a Body exists but no bytes are read, the logged Body can be an empty string.

## Trace ID

By default, Log Mask reads the first non-empty value from the MDC keys `traceId` and `trace_id`, in that order:

| Property | Default | Effect |
| --- | --- | --- |
| `log-mask.logging.rest-template.trace-id.enabled` | `true` | Whether to read a Trace ID from MDC |
| `log-mask.logging.rest-template.trace-id.mdc-keys` | `[traceId, trace_id]` | Keys to check in order; the first non-empty value is written to the log |

```yaml
log-mask:
  logging:
    rest-template:
      trace-id:
        enabled: true
        mdc-keys:
          - traceId
          - trace_id
```

Log Mask only reads an existing Trace ID. It does not create or modify MDC. When no value is available, `traceId` is `null` in the log.

## Custom masking strategies

`MaskTypeDefinition` beans in the application are registered automatically when the Starter initializes. Path, Query, and Header rules reference them through `type-code`; Body fields reference them through `@Mask`.

See [Custom masking strategies](../log-mask-core/README.md#custom-masking-strategies) in the Core guide for definition and registration instructions.

## FAQ

### Why is there no call log?

Check these conditions in order:

1. The RestTemplate is selected by annotation, Bean name, or code.
2. `log-mask.logging.rest-template.enabled` is `true`.
3. The `log.mask.http` logger accepts `INFO`.
4. `observedInstanceCountAtStartup` is greater than 0 in the startup log.
5. `exchangeEventsEnabledAtStartup` is `true` in the startup log.

### Does Log Mask process every RestTemplate automatically?

No. Call logging is installed only on a RestTemplate selected by annotation, Bean name, or code.

### Why are fields inside a JSON String Body not masked?

A String Body is logged as an ordinary string and is not parsed as JSON again. For field-level masking, use a specific Java type with `@Mask` or `@LogExclude`.

### Why is Body an empty string?

Check the corresponding `bodyState`:

- `DISABLED`: Body logging is disabled.
- `LIMIT_EXCEEDED`: Body exceeded the maximum log size.
- `PROCESSING_FAILED`: Body log generation failed.

An error response can contain a Body while the business flow reads no bytes. In that case, `bodyState` can still be `SUCCESS` while Body is an empty string.
